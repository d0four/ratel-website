03
03 Anatomy of a mission

One run, start to merge.

mission graph01 / plan
Stage 01 / plan

It reads your repo, then argues with your spec.

The orchestrator takes your goal and turns it into a contract: the exact, testable definition of done. Vague in, questions back. It will not start building against a spec it can already break.

→ goal becomes 9 acceptance criteria
→ split into 3 Gherkin features
→ you approve the plan, then step out
Stage 02 / dispatch

One job each. No shared mess.

Every feature gets its own worker, its own branch, and a fresh context with nothing else in it. Tests first, then code, then a handoff you can parse. Nothing steps on anything else.

→ feat/F1 · feat/F2 · feat/F3
→ test-driven, integrated in order
→ no shared state left to corrupt
Stage 03 / judge

A second agent tries to break the first.

Read-only reviewers go hunting. Lint, types, a hostile code review, and a user-testing pass that drives the live app across parallel browser shards. They rank what they find by how much it hurts.

→ scrutiny + independent code review
→ 3 user-test shards in parallel
→ findings ranked by severity
Stage 04 / gate

Plain code decides. Not a model.

A deterministic gate reads the verdict. One high-severity issue, or a handoff that does not parse, and nothing merges. The agent that wrote the code gets no vote on whether it ships.

→ zero high-severity required
→ handoff must parse clean
→ integrated, then validated
Stage 05 / ship

You come back to a branch, not a mess.

Everything that passed is merged and reviewable. The mission stops only on a real blocker, and tells you exactly why. Stepping away cost you nothing but the time it took.

→ merged and ready to read
→ full event log written to disk
→ one review and you are done
04
04 The split

Four roles. None of them trust each other.

A human used to plan the work and judge the result. Ratel keeps those two jobs in different hands, so no agent ever grades its own homework.

01
Orchestrator

The boss

sits in your chair

Scopes the work, interrogates the spec, writes the contract, and calls every shot for the whole run.

02
Workers

The crew

heads down, one job

One feature, one branch, one clean mind. Test-driven, isolated, and done when the handoff parses.

03
Validators

The skeptic

read-only, out for blood

Reviews, scrutiny, and live user tests built to find what the crew missed. Hostile on purpose.

04
Gates

The bouncer

deterministic, unmovable

Hand-written code, not a model. No green checks, no entry. Nothing talks its way past the door.

05
05 Under the hood

Boring on purpose.

The interesting part of an autonomous run is everything that does not go wrong. These are the parts that keep it that way.

Gates that cannot be skipped

A feature merges only on clean checks and a handoff that parses. No exceptions in the code.

Budgets it respects

Tokens, dollars, wall-clock, and run count checked every turn. It stops before it overspends.

Failover that holds

A retryable error reroutes through a circuit breaker instead of killing the whole run.

A window into the run

Every span, tool call, diff, and halt streams to a dashboard while it works.

Branches kept apart

Each feature lands on its own branch and integrates in order, never on top of broken work.

Handoffs you can read

Workers report what they finished, what they skipped, and what they hit, as parsed JSON.

Testing that fans out

User flows split across parallel shards, so coverage scales instead of dragging.

Errors it fixes first

Compile, lint, and runtime failures get a correction pass before anything reaches the gate.

06 The CLI

Everything you will type.

A handful of commands, no learning curve. Install once, then drive whole missions from one line.

Commands

ratel mission start "<spec>"Plan, build, review, and gate a whole feature set from one prompt.
ratel mission statusSee exactly where the current run is, phase by phase.
ratel mission resume <id>Pick a paused mission back up after you unblock it.
ratel observatoryOpen the live dashboard for every agent, call, and gate.

Install

pi install @ratel/pi-extensionWire Ratel into pi as an extension.
ratel plugin opencodeDrop-in plugin for an opencode setup.
ratel run --directRun a mission with no host agent at all.

09
09 Where the line sits

Rigid where it must be.
Thinking where it has to.

An agent left to manage itself drifts. Ratel draws a hard line between what a model is allowed to decide and what only code decides. Nothing lives in the gap.

01 / STRUCTURE

Code owns the spine

Schemas, branches, timeouts, routing, and merge logic are written by hand and never negotiable. No model gets to decide them mid-run.

02 / JUDGMENT

Models own the work

Planning, building, testing, and review are where thinking belongs. The agents do the hard part inside walls that hold.

03 / TRUST

Gates own the merge

Code ships only past checks no agent can move. The floor stays fixed, which is the only reason walking away is safe.

10
10 Straight answers

No marketing. Just the honest version.

✓ Built for

  • Shipping a v1 from a spec you can actually write down.
  • Well-scoped features with clear acceptance criteria.
  • Migrations and refactors with a defined target.

— Not for

  • Vague ideas with no definition of done.
  • One-line tweaks you would finish faster by hand.
  • Work you cannot describe to another engineer.
Is it actually autonomous, or do I babysit it?
You approve the plan once, at the start. After that it runs on its own and stops only on a real blocker. The one human checkpoint is the plan, because a wrong plan wastes the whole run. Everything after is hands-off.
What does a run cost?
Whatever your model charges, capped. You set token, dollar, and time limits up front, and it stops before it crosses them. No surprise bill at the end of a long run.
Is this a demo, or does it ship real code?
It works on real repos, in git, behind gates. Point it at narrow, well-tested work first. The tighter your spec, the better the result. Vague in, vague out.
Which models can I use?
Yours. It runs on pi or opencode and uses your keys. No model is bundled and nothing is locked in. Swap whenever you want.
Where does my code go?
Nowhere. It runs where you run it, on your machine, with your keys. Your code never leaves your environment.
What happens when it gets stuck?
It stops, writes why to the log, and waits. You read a clear reason, not a wall of broken files. Then you unblock it and it keeps going.
